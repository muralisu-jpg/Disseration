[iter 04] REVERTED the iter-3 IBEFORE rule (`if is_perfect(w1) and not is_perfect(w2): return "IBEFORE"`) — it overfit the example and REGRESSED the hidden test (IBEFORE 0.45->0.438, BEFORE 0.728->0.724, macro_f1 0.145->0.144). THEN made one focused IDENTITY change: in same_event(), when BOTH mentions are multi-word spans (>=4 content tokens each, i.e. two distinct quoted utterances), a single shared content lemma is no longer enough — such spans routinely share incidental words ("get"/"drowned"/"fish") yet are different events. Now require near-identity (token-set Jaccard >= 0.8) for the both-multi-word case; short mentions still use the truncation-tolerant single-lemma match (so one-side-quote IDENTITY like "Hea"/"heate", "tur"/"Tur" is preserved). Empirical (example): IDENTITY P 0.973->1.000, F1 0.935->0.947 (R held 0.900, removed the 1 false positive pair where two different quotes shared words). Bonus: that pair now correctly falls through to the both-speech IBEFORE rule, IBEFORE F1 0.449->0.480. No relation dropped. Overall acc 0.7585->0.7633, macro_f1 0.4314->0.4362. Generalizes: distinct long quotes sharing incidental words are no longer mislabeled IDENTITY (truncation-safe; uses no story-specific text).
[iter 03] IBEFORE recall boost. NEW rule (priority 3b, after IDENTITY and both-perfect SIMULTANEOUS): if e1 is past-perfect ("had/has/have" + verb) and e2 is NOT perfect, label IBEFORE — a completed anterior event meets the following simple-past event immediately (e.g. "had passed by" -> hobbled, "had finished" -> said, "had gnawed" -> fell down). Found empirically: among single-perfect-e1 example pairs, gold is IBEFORE 4/5 (only "had heard"->longed is BEFORE). The cue word had/has/have is short and never truncated, so it generalizes. The previous IBEFORE rule (both events speech) had recall only 0.31 (precision 0.79); this adds 4 true positives at cost of 1 false positive. Did NOT loosen to e2=reporting-verb (would have added 4 BEFORE false positives, net neutral). Example self-eval: IBEFORE F1 0.449->0.556 (R 0.314->0.429, P 0.789 held); BEFORE F1 also 0.795->0.803 (rule removed 4 BEFORE false-positives, lost only 1 true positive); IDENTITY/SIMULTANEOUS/AFTER unchanged; acc 0.758->0.773, macro_f1 0.435->0.448.
[iter 02] SIMULTANEOUS + AFTER refinement. (a) NEW SIMULTANEOUS rule: if BOTH events are perfect aspect ("had/has/have" + verb), label SIMULTANEOUS — two anterior background events presented together are temporally indistinguishable (e.g. "had heard"/"had cried"). Distinguishes from IBEFORE/AFTER which have only ONE perfect side. (b) DROPPED the iter-1 reversed-text-order rule (o2<o1 -> AFTER): the guideline says "maintain links forward" / keep subject e1 before object e2 in text, so genuine AFTER has o1<o2; the reversed rule was backwards, never fired on the example, and produced 10 all-wrong AFTERs on the test set (AFTER F1 0.000). AFTER now relies only on the clean anterior signal (e2 = "having..." perfect participle). Example self-eval: SIMULTANEOUS F1 0.67->0.74 (R 0.50->0.58, P 1.00 held), AFTER F1 1.00 held, BEFORE/IDENTITY/IBEFORE unchanged, macro_f1 0.427->0.435.
[iter 01] Built initial classifier (classify.py): infers tense/signals from the truncated `word` string + offsets (no tense field at test time). Rules in priority: (a) same-offset INFINITIVE list -> SIMULTANEOUS; (b) shared content lemma -> IDENTITY; (c) e2 "having..." perfect participle -> AFTER; (d) reversed text order (o2<o1) -> AFTER; (e) both events speech/reporting/quote -> IBEFORE; (f) default BEFORE. Targets IDENTITY, AFTER, IBEFORE, SIMULTANEOUS to beat the BEFORE-everywhere floor. Example self-eval: acc 0.754, macro_f1 ~0.43 (AFTER 1.00, IDENTITY 0.94, SIMULTANEOUS 0.67, IBEFORE 0.45, BEFORE 0.79; ENDED_BY/INCLUDES/IS_INCLUDED/BEGINS still 0 — left for later iterations).
